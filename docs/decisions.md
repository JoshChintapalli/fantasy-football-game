# Engineering and Architecture Decisions
**Python/Streamlit over a Web Framework**
- Context: Needed a frontend for a data-heavy game with real historical NFL stats.
- Decision: Python with Streamlit for both backend logic and UI.
- Considered Alternatives: React frontend with a Flask/FastAPI backend.
- Reasoning: Streamlit allows Python to handle both data manipulation and UI in one language, allowing the project to be completed without having to learn new text stacks and frameworks, and reducing complexity for an early-stage project. 
- Drawbacks: Limited UI customization compared to a full frontend framework; visual wheel and automated animations will require additional effort or libraries.

**CSV over Database**
- Context: Needed persistent storage for historical NFL player data.
- Decision: Store and filter data in a local CSV file generated by "data_pipeline.py".
- Considered Alternatives: SQLite or PostgreSQL database.
- Reasoning: The dataset is read-only, has no relational structure, and is generated once from a known source. A database would add overhead infrastructure with no meaningful benefit for the project scale.
- Drawbacks: Slower initial load on first run; anyone cloning the GitHub repo must run data_pipeline.py to generate the CSV file.

**DST and K Deferred from Roster**
- Context: Original plan featured a 9-player roster including DST and K positions.
- Decision: Launched MVP with 7 player roster (QB, RB1, RB2, WR1, WR2, TE, FLEX).
- Considered Alternatives: Hardcoding data or generating placeholder data for K and DST positions.
- Reasoning: "nfl_data_py" does not include stats for DST or K. Compiling and manually adding historical data would be extremely and unrealistically time consuming; generating placeholder data would undermine the effort to use real, historical NFL data and make the game unrealistic. Deferred until a reliable data source is identified.
- Drawbacks: Roster does not match original plan; K and DST addition is tracked in "roadmap.md" as a future feature.

**Named Profiles over Full Authentication**
- Context: Persistent tracking required distinguishing between different users.
- Decision: Use name-based profiles stored locally rather than full login/password system.
- Considered Alternatives: JWT authentication, OAuth (Sign in with Google).
- Reasoning: The dataset being tracked is small and single-player. Full authentication carries real security risks if implemented incorrectly (password hashing, session management, account recovery). A named profile approach delivers per-person tracking without complexity and security concerns.
- Drawbacks: No cross-device sync; profiles are local only.

**Functional Design Before OOP**
- Context: "game_engine.py" was built using standalone functions with dictionaries before class structure was considered.
- Decision: Build with functions first, refactor to OOP after MVP completion with pytest as a safety net.
- Considered Alternatives: Designing backend logic using OOP from the beginning
- Reasoning: Plans for how to structure backend logic was still unclear at the start of the project and many bugs and edge cases with FLEX rules and the bonus round emerged during building. Committing to OOP would have likely caused restructuring of the class design itself, whereas functions were easier to reshape mid-discovery of a bug. 
- Drawbacks: Many separate session state variables in "app.py" instead of one object; OOP refactor was planned later as an engineering improvement, but requires entire restructuring of "game_engine.py", risking logic breaking.

**Session State CSV Loading Instead of Caching Data**
- Context: "app_data.csv" needs to be loaded once and available throughout the session.
- Decision: Load CSV into a Streamlit session state variable using the existing session state initialization pattern.
- Considered Alternatives: Using Streamlit's "@st.cache_data" decorator, the idiomatic approach.
- Reasoning: Session State approach was implemented before "st.cache_data" was introduced into the project. Sessions state approach works correctly as far as functionality is concerned.
- Drawbacks: Not idiomatic Streamlit; refactor to st.cache_data is tracked in "roadmap.md" as an engineering improvement.
